# Day 9 — Token counting & cost tracking

**Objective.** Know the token count and dollar cost of every call.

**Outcome.** Each response logs prompt/completion tokens and estimated cost;
a running total per `request_id` is queryable via HTTP.

## What we built

- **`coursesmith/infrastructure/shared/utils/usage_tracker.py` — `UsageTracker`.**
  In-memory accumulator keyed by `request_id`. Exposes:
  - `record(prompt, completion, cost)` — reads `request_id` from the structlog
    contextvar (set by Day 8's `LoggingMiddleware`) and accumulates into the
    bucket for that request.
  - `snapshot(request_id)` — returns the `UsageModel` for that request, or
    `None` if no LLM calls were recorded for it.
  - `get_usage_tracker()` — `@lru_cache`'d zero-arg factory; effectively a
    singleton accessor so every adapter instance shares the same store.
- **`LiteLlmAdapter` records cost on every LLM call.** Both `complete()` and
  `stream()` route through a single private `_log_llm_costs(messages, result)`
  that:
  - Counts prompt tokens with `token_counter(model, messages)`.
  - Counts completion tokens with `token_counter(model, [result.choices[0].message])`.
  - Computes per-token costs with `cost_per_token(model, prompt_tokens,
    completion_tokens)`.
  - Computes total completion cost with `completion_cost(result)`.
  - Logs `llm_costs_calculated` with all five numbers (already tagged with
    the current `request_id` thanks to `merge_contextvars`).
  - Calls `usage_tracker.record(...)` so the same numbers roll up into the
    request's bucket.
- **Streaming gets the same treatment.** The provider doesn't include usage
  in stream chunks by default, so `stream()` passes
  `stream_options={"include_usage": True}` to `acompletion(...)` and
  accumulates chunks while yielding them. When the stream ends, the chunks
  are rebuilt into a non-stream-shaped response via
  `litellm.stream_chunk_builder(chunks=..., messages=...)` and fed into the
  same `_log_llm_costs` helper. `stream_chunk_builder` returns
  `ModelResponse | TextCompletionResponse | None`; we narrow with
  `isinstance(..., ModelResponse)` so the chat-completion path runs and the
  edge cases (text-completion, empty stream from a client disconnect) are
  silently skipped — nothing meaningful to record.
- **`GET /courses/usage/{request_id}` endpoint.** Returns the
  `UsageModel` (prompt tokens, completion tokens, cost USD) for a given
  request_id. Returns HTTP 404 with `null` body for unknown ids so
  "no record" is distinguishable from "request that did zero work."

## Why this satisfies the spec

- **"Know the token count and dollar cost of every call."** Every LLM call —
  `complete()` *and* `stream()` — runs through `_log_llm_costs`, which emits
  prompt tokens, completion tokens, per-token costs, and total completion
  cost on a single structured event.
- **"Each response logs prompt/completion tokens and estimated cost."** The
  `llm_costs_calculated` log line carries `tokens_prompt`,
  `tokens_completion`, `token_cost_prompt`, `token_cost_completion`, and
  `completion_cost`.
- **"A running total is queryable."** Per the Day 9 brief's most-direct hint
  ("accumulate per request_id so a later multi-call agent run rolls up to one
  number"), the running total is *per-request*, not process-wide.
  `GET /courses/usage/{request_id}` returns it. A multi-call agent run that
  shares one `request_id` will accumulate every call into the same bucket
  because `record()` reads the `request_id` from the structlog contextvar at
  call time — not from a parameter — so `complete()`, `stream()`, and any
  future inner LLM calls within the same request all land in one place.

## How to verify

1. **Start the API**:

   ```sh
   uv run fastapi dev coursesmith/app.py
   ```

2. **Fire a request with a known `X-Request-ID`**:

   ```sh
   curl -X POST http://localhost:8000/courses \
        -H 'Content-Type: application/json' \
        -H 'X-Request-ID: demo-9' \
        -d '{"topic": "Two-day intro to git"}'
   ```

3. **Check the cost log line** in the server output — look for an event
   named `llm_costs_calculated` carrying `tokens_prompt`,
   `tokens_completion`, `completion_cost`, and `request_id=demo-9`.

4. **Query the running total**:

   ```sh
   curl http://localhost:8000/courses/usage/demo-9
   ```

   Returns the per-request totals as JSON.

5. **Unknown request_id** returns HTTP 404 with `null`:

   ```sh
   curl -i http://localhost:8000/courses/usage/never-happened
   ```

6. **Streaming path also records cost** (after the stream completes):

   ```sh
   curl -N -X POST http://localhost:8000/courses/stream \
        -H 'Content-Type: application/json' \
        -H 'X-Request-ID: demo-9-stream' \
        -d '{"topic": "..."}'
   curl http://localhost:8000/courses/usage/demo-9-stream
   ```

## Design notes

- **Why per-request keying, not a process-wide counter?** The brief's most-
  direct hint calls for accumulating per `request_id` so a multi-call agent
  run rolls up to one number. That's the load-bearing use case. A global
  counter is a different feature (process-lifetime spend monitoring) and
  would need persistence anyway to be meaningful across restarts — a
  separate day's problem.
- **Why does `record()` pull `request_id` from the contextvar instead of
  taking it as an argument?** Because the LLM adapter doesn't (and
  shouldn't) know about inbound HTTP correlation. Day 8 already binds
  `request_id` into the structlog contextvar at the middleware boundary;
  reading it back here keeps the adapter free of transport concerns and
  ensures every call within a single request lands in the same bucket
  automatically, no matter how many layers deep it's made from.
- **Why one private `_log_llm_costs` for both `complete()` and `stream()`?**
  Removes duplication and guarantees both paths log the same shape and
  record into the tracker identically. Adding a new metric (e.g. latency
  per call) is a one-line edit, not two.
- **Why `stream_options={"include_usage": True}` matters.** Without it, the
  streamed chunks carry no usage block, and `stream_chunk_builder(...)`
  produces a response whose `.usage` is empty — `completion_cost(...)`
  then returns 0.0 and you'd silently log zero cost on every stream.
- **In-memory and process-local.** Restarting the server clears the store.
  Acceptable for the challenge (and matches the "queryable" framing — the
  data is queryable while the process lives). Persistence (sqlite,
  Prometheus counter, Redis) is a future-day concern; the `UsageTracker`
  is small enough to swap in place when that day comes.
- **404 vs zeros for unknown ids.** `snapshot()` returns `None` for an
  unknown id, and the endpoint maps that to HTTP 404. Honest: "we have no
  record of this" is genuinely different from "this request ran but spent
  nothing," and the wire protocol shouldn't conflate them.
