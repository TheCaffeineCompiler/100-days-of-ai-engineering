# Day 11 — Define a tool schema

**Objective.** Describe a tool to the model in JSON-schema form.

**Outcome.** A `get_current_time` tool defined with name, description, and
a JSON schema for its parameters — ready to pass to the model. The handler
and the schema share a single Pydantic args model so neither can drift
from the other.

## What we built

- **`coursesmith/use_cases/create_course_outline/tools/get_current_time_tool.py`** —
  three things in one short module:
  - `GetCurrentTimeParams(BaseModel)` — the args schema. One field,
    `pattern: str = "%Y-%m-%dT%H:%M:%S"`. Source of truth for the
    parameter shape, the default, and runtime validation.
  - `get_current_time_schema()` — returns the tool definition in the
    OpenAI/LiteLLM `{"type": "function", "function": {...}}` shape.
    `parameters` is `GetCurrentTimeParams.model_json_schema()` passed
    straight through — no hand-written `properties`/`required`/`type`
    blob to drift from the model.
  - `get_current_time(pattern=...)` — the handler. Returns
    `datetime.now().strftime(pattern)`.
- **Six unit tests** at
  `tests/use_cases/create_course_outline/tools/test_get_current_time_tool.py`
  pinning the contract: top-level schema shape, parameters are a valid
  JSON-schema object, the default is exposed to the model, the function
  default and the Pydantic default agree, and the handler round-trips
  through `strftime`/`strptime` for both the default and a custom
  pattern.

## Design notes

- **Why a `tools/` folder under a use case, not a `ToolsPort`?** The
  other shared ports (`PromptsPort`, `LlmPort`) are *behavioral*
  contracts — many implementations could satisfy them. A single tool's
  schema isn't an interface, it's a value: one schema + one handler
  bundled together. Premature port-ification would invent an
  abstraction that has no second implementation. The natural port
  (`ToolsPort.invoke(name, args)` plus a registry of schemas) shows up
  when the agent loop arrives in Day 12+; today's tool drops straight
  into that registry without refactoring.
- **Why one Pydantic model for both schema and validation?** Two
  hand-written shapes diverge silently — the schema says one thing,
  the handler accepts another, and the bug only surfaces when the
  model passes args the handler can't take. `model_json_schema()`
  emits a valid JSON-schema object (with `type: object`, `properties`,
  `required`, and `default`s baked in) directly from the same class
  that will validate the call. One source of truth.
- **Why test that the two defaults match?** Today the default
  `"%Y-%m-%dT%H:%M:%S"` lives on both `GetCurrentTimeParams.pattern`
  and `get_current_time(pattern=...)`. The test is a tripwire — if
  one moves and the other doesn't, CI fails before the divergence
  ships. The cleaner fix is to drop the function-signature default
  and call the handler through the model (`GetCurrentTimeParams(**kwargs)`
  → `strftime(params.pattern)`); that test goes away the day the
  duplication does.
- **Description is for the model, not the human.** Per the brief's
  third hint: written as a prompt fragment ("Call this tool to get the
  current time formatted with the given strftime pattern."). The
  default is already in the schema as `"default": "%Y-%m-%dT%H:%M:%S"`,
  so the description doesn't restate it.

## How to verify

1. **Run the new tests in isolation**:

   ```sh
   uv run pytest tests/use_cases/create_course_outline/tools -v
   ```

2. **Eyeball the schema** the model would see:

   ```sh
   uv run python -c "import json; from coursesmith.use_cases.create_course_outline.tools.get_current_time_tool import get_current_time_schema; print(json.dumps(get_current_time_schema(), indent=2))"
   ```

   Confirms `parameters` is a real JSON-schema object (`type`,
   `properties`, `default`) — not a stringified blob.

3. **Call the handler directly**:

   ```sh
   uv run python -c "from coursesmith.use_cases.create_course_outline.tools.get_current_time_tool import get_current_time; print(get_current_time())"
   ```
