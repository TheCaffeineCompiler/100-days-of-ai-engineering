# Day 8 — Structured logging

**Objective.** Replace prints with structured, correlatable logs.

**Outcome.** Every request logs JSON with a `request_id`; you can grep one
request end to end.

## What we built

- **`coursesmith/config/logging_config.py`** — single entry point
  `configure_logging(json_logs, log_level)` that wires structlog and the
  stdlib `logging` module to the *same* output pipeline. Without the bridge,
  third-party libraries (LiteLLM, uvicorn, httpx) that use the stdlib
  `logging` API would emit in their own format alongside structlog's JSON.
  The bridge is the canonical structlog pattern:
  - Structlog's processor chain ends in
    `ProcessorFormatter.wrap_for_formatter`, handing off to stdlib.
  - The root `logging` handler uses `ProcessorFormatter(processor=renderer,
    foreign_pre_chain=shared_processors)` so stdlib records also run through
    `merge_contextvars`, `add_log_level`, `TimeStamper`, etc. before the
    final JSON (or pretty) renderer.
- **`merge_contextvars` as the first processor.** Structlog's
  `contextvars` module reads from a contextvar at *log time* and merges its
  contents into the event dict. So once middleware binds `request_id`,
  every subsequent log line — including stdlib records from LiteLLM — picks
  it up automatically. No per-call `.bind(request_id=...)`.
- **`coursesmith/infrastructure/adapters/inbound/rest/middleware.py` —
  `LoggingMiddleware`, raw ASGI** (not `BaseHTTPMiddleware`). Binds
  `request_id` (from `X-Request-ID` header, or a freshly minted UUID4) into
  the structlog contextvar, logs `request_started`, awaits the downstream
  app, logs `request_completed` (or `request_failed` on exception) with
  `status_code` and `duration_ms`, and echoes `X-Request-ID` back on the
  response by intercepting the ASGI `http.response.start` message. Non-HTTP
  scopes (lifespan, websockets) pass through untouched.
- **LiteLLM's own handlers cleared.** LiteLLM attaches its own
  `StreamHandler` (plain-text, ANSI-coloured) to its loggers at import time.
  `configure_logging` calls `.handlers.clear()` + `propagate=True` on
  `LiteLLM`, `LiteLLM Router`, and `LiteLLM Proxy`, so their records flow
  *only* through root → JSON formatter. One output per event, in our format,
  carrying the request's `request_id`.
- **Two new settings.** `LOG_JSON_ENABLED` (bool) and `LOG_LEVEL` (the
  `LogLevel` `Literal` from `logging_config.py`). Wired through
  `Settings`, called from `app.py` *before* `FastAPI()` is constructed so
  the framework's own startup logs are already JSON.

## Why this satisfies the spec

- **"Replace prints with structured, correlatable logs."** No `print`
  statements anywhere; every log line is a JSON object with consistent
  fields (`event`, `level`, `timestamp`, plus whatever the caller bound).
  Correlation is provided by `request_id` flowing through `merge_contextvars`.
- **"Every request logs JSON with a `request_id`."** Verified live: hit
  `POST /courses`, every line emitted during the request — middleware's
  `request_started`/`request_completed`, LiteLLM Router's "Routing
  strategy", LiteLLM's `acompletion` 200 OK — all share the same
  `request_id`.
- **"You can grep one request end to end."**
  `cat logs | grep <request_id>` returns every line for that request and
  nothing else.

## How to verify

1. **Start the API**:

   ```sh
   uv run fastapi dev coursesmith/app.py
   ```

2. **Fire a request with a known `X-Request-ID`** so you control the
   correlation key:

   ```sh
   curl -X POST http://localhost:8000/courses \
        -H 'Content-Type: application/json' \
        -H 'X-Request-ID: demo-123' \
        -d '{"topic": "Two-day intro to git"}'
   ```

3. **Grep the server output** for `demo-123` — you should see the full
   timeline: `request_started`, LiteLLM router routing-strategy line,
   LiteLLM completion-call line, LiteLLM HTTP-status line,
   `request_completed`, all tagged with `"request_id": "demo-123"`.

4. **Verify the response carries it back**:

   ```sh
   curl -i -X POST http://localhost:8000/courses \
        -H 'Content-Type: application/json' \
        -H 'X-Request-ID: demo-123' \
        -d '{"topic": "..."}' | grep -i x-request-id
   ```

5. **Streaming endpoint still streams**:

   ```sh
   curl -N -X POST http://localhost:8000/courses/stream \
        -H 'Content-Type: application/json' \
        -d '{"topic": "..."}'
   ```

   Tokens arrive incrementally — raw ASGI middleware doesn't buffer
   responses the way `BaseHTTPMiddleware` does, so Day 5's SSE behaviour is
   intact.

## Design notes

- **Why raw ASGI middleware, not `BaseHTTPMiddleware`?** Starlette's
  `BaseHTTPMiddleware` wraps the downstream app in a child anyio task. In
  practice, contextvars bound in `dispatch` aren't reliably visible to
  log records emitted from the endpoint task — there's a known interaction
  between Starlette's cancel-scope wrapping and contextvar inheritance.
  Raw ASGI runs the downstream app in the same task scope as the
  binding, so `merge_contextvars` finds `request_id` on every record.
  Bonus: no response buffering, so SSE still streams.
- **Why bridge stdlib logging instead of monkey-patching libraries?** A
  bridge is the only sustainable strategy. Every dependency we add brings
  its own logger; the bridge means we don't need to know or care.
- **Why clear LiteLLM's handlers in `configure_logging`, not in
  `lite_llm_adapter.py`?** The handlers are attached at LiteLLM import
  time, globally on the named loggers — it's a logging-system concern,
  not an adapter concern. The same trick would apply to any future noisy
  third-party logger (`uvicorn.access` is the usual next offender).
- **Why `LOG_JSON_ENABLED` instead of just always-JSON?** Local dev
  benefits from `ConsoleRenderer(colors=True)` — pretty-printed,
  human-scannable. Production wants JSON for ingest by Loki/CloudWatch/etc.
  One flag, no two-codebase logging.
- **Why a typed `LogLevel = Literal[...]`?** A typo (`LOG_LEVEL=info` —
  lowercase) should fail at `Settings()` construction, not at the first
  log call. The `Literal` makes pydantic-settings reject anything that
  isn't one of the five canonical level names at startup.
