# Day 13 — Execute the tool & feed the result back

**Objective.** Close the loop on a single tool: model asks → you run it
→ model answers.

**Outcome.** The final assistant message incorporates the tool's real
output. A topic that asks for the current date/time now returns a
populated `CourseOutline` with the actual time baked in, instead of
the Day 12 sentinel.

## What we built

- **`tools/__init__.py` becomes a tiny tool registry.**
  - `get_tools()` returns the list of tool schemas to pass to the
    LLM (today: just `get_current_time_schema()`).
  - `execute_tool(name, **kwargs)` dispatches to the actual handler by
    name. If a tool raises, the exception is caught and the error
    string is returned as the tool's "output" — the model can read
    that and recover or apologise.
- **`CourseOutlineService.create()` is now an agent loop**, not a
  single call:

  ```text
  for at most max_steps iterations:
      call LLM with current messages + tools
      if response has tool_calls:
          append the assistant message (which carries the tool_calls)
          for each tool_call:
              run it, append a role="tool" message with tool_call_id + result
          → next iteration
      else:
          return CourseOutline.model_validate_json(content)
  if loop budget exhausted:
      return the Day 12 sentinel as a last-resort fallback
  ```

  The protocol bit that matters: the assistant message containing
  `tool_calls` is appended *before* the tool reply messages, and each
  tool reply carries the matching `tool_call_id` so the model can
  correlate them. Without that pairing the model silently ignores the
  tool result and re-asks for the same call.
- **`max_steps = 5`.** Cheap insurance against runaway loops. Today's
  single-tool case completes in one round-trip; the budget is just
  slack.

## Why we used `model_validate_json` only on the no-tool branch

`response_format=CourseOutline` is still passed on every iteration —
once the model decides to return JSON (the tool branch is done), it'll
honour the schema. While the model is in tool-using mode it returns
`content=None` and we never try to validate it. The condition
`if tool_calls: …  else: return validate(content)` is exactly the
discriminator between the two response shapes.

## Known smells we left for later

- **`except Exception` in `execute_tool` is too wide.** Returning an
  error string back to the model is the right shape (the model can
  recover), but `Exception` swallows real bugs too — typos in handler
  logic, `AttributeError` from refactors. The narrower form is to
  catch only the exceptions a specific tool can legitimately raise.
  Deferred until we have more than one tool.
- **`messages.append(result.choices[0].message)` pushes a LiteLLM
  `Message` *object* into a `list[dict[str, str]]`.** LiteLLM accepts
  it at runtime, but the typed signature on the local variable lies.
  Fix is one of: widen the local list type, or convert with
  `.model_dump()` before appending. Deferred until it actually bites.
- **`if name == "get_current_time"` dispatch in `execute_tool`** —
  one-tool registry. The dict-of-handlers pattern is the obvious
  refactor when tool #2 arrives, not before.
- **Loop-exhausted sentinel.** Still
  `CourseOutline(title="n/a", day_items=[])` if `max_steps` runs out.
  A real impl would raise a typed error and the inbound layer would
  translate it; that's a Day 6-style error-handling task, not Day 13.

## How to verify

1. **Start the API**:

   ```sh
   uv run fastapi dev coursesmith/app.py
   ```

2. **Send a topic that requires the time tool**:

   ```sh
   curl -X POST http://localhost:8000/courses \
        -H 'Content-Type: application/json' \
        -d '{"topic": "AI engineering for backend developers. Add current date time as creation date of the course."}'
   ```

3. **Confirm in the response** that:
   - the `title` and `day_items` are real content (no sentinel),
   - the actual current date/time appears somewhere in the body
     (typically in the title or first day's objective).

4. **Confirm in the logs** you see two `litellm.acompletion ... 200 OK`
   lines for a single request, sandwiching one `call_function` event
   for `get_current_time`. That's the closed loop.
