# Day 3 ā€” Prompt templating & versioning

**Objective.** Move prompts out of f-strings into versioned, testable templates.

**Outcome.** Prompts live in files keyed by name + version; swapping a version
doesn't require touching call sites.

## What we built

- **Prompt file registry** under `resources/prompts/<name>/v<version>.prompt.txt`.
  First entry: `resources/prompts/course_outline/v1.prompt.txt`, containing the
  template that used to be a module-level `_prompt` string in the service.
  Versions are sibling files (`v2.prompt.txt`, `v3.prompt.txt`, ā€¦); the
  directory groups all versions of one prompt.
- **`PromptsPort`** (`coursesmith/use_cases/shared/ports/prompts_port.py`) ā€”
  the abstract interface use cases depend on:

  ```python
  class PromptsPort(ABC):
      @abstractmethod
      def get_prompt(self, name: str, version: int) -> str: ...
  ```

- **`PromptsAdapter`** (`coursesmith/infrastructure/shared/adapters/prompts_adapter.py`) ā€”
  the file-backed implementation. Resolves a `(name, version)` pair to
  `<base_path>/<name>/v<version>.prompt.txt` and returns its contents.
- **Hexagonal layout introduced.** The package now has two top-level layers
  under `coursesmith/`:
  - `use_cases/` for application logic (services, ports, models),
  - `infrastructure/` for adapters that implement the ports.

  Shared cross-feature pieces live under `ā€¦/shared/`. `CourseOutlineService`
  depends on the `PromptsPort` interface, not the file-system adapter ā€” wiring
  happens in `hello.py`.
- **`coursesmith/__init__.py`** exports a `RESOURCES_DIR` constant computed
  from `__file__`, so the prompts directory is discoverable without env vars
  and without entry points having to know the repo layout.
- **`Settings`** (`coursesmith/settings.py`) ā€” `pydantic-settings`
  `BaseSettings` with no defaults. Missing env vars now raise a
  `ValidationError` at startup instead of silently substituting empty strings.
  Fields: `litellm_model`, `litellm_api_key`, `course_outline_prompt_version`.
- **`CourseOutlineService` refactor** ā€” constructor takes a `PromptsPort` and
  a `prompt_version`; the prompt *name* is a class constant (`PROMPT_NAME =
  "course_outline"`) because the name is owned by the service, not by config.
  Bumping versions is a `.env` edit; nothing in `.py` files changes.
- **Pytest wired up.** `pyproject.toml` adds `pytest` to the dev group, plus a
  `[tool.pytest.ini_options]` block (`pythonpath = ["."]`, `testpaths =
  ["tests"]`) so test collection finds the `coursesmith` package without
  needing a build backend.
- **First test:** `tests/infrastructure/shared/adapters/test_prompts_adapter.py`
  asserts that the committed `course_outline` v1 file contains the `{topic}`
  placeholder ā€” a contract test between the adapter's consumers and the
  on-disk template.
- **Mypy + pydantic plugin.** `pyproject.toml` now sets
  `plugins = ["pydantic.mypy"]`, which teaches the type checker that
  `BaseSettings` fields without defaults are populated from the environment.
  Removed the static-analysis warning about `Settings()` "missing arguments"
  without resorting to `# type: ignore` comments.

## Why this satisfies the spec

- "**Prompts live in files/registry keyed by name+version**" ā€” files at
  `resources/prompts/<name>/v<version>.prompt.txt`, fronted by `PromptsPort`
  as the interface and `PromptsAdapter` as the file-system implementation.
- "**Swapping a version doesn't require touching call sites**" ā€” the service
  asks the port for `(PROMPT_NAME, version)`. Set `COURSE_OUTLINE_PROMPT_VERSION=2`
  in `.env`, drop a `v2.prompt.txt` next to `v1.prompt.txt`, re-run. No `.py`
  edits.
- "**Testable templates**" ā€” the adapter is plain I/O against a
  `Path`-typed base directory, so tests can point it at the real
  `resources/prompts/` (contract tests) or at `tmp_path` (isolated unit
  tests).

## How to verify

1. **Default version works**:

   ```sh
   uv run python -m coursesmith.hello "AI engineering for backend developers"
   ```

2. **Version swap works without code edits.** Create a v2 template and bump
   the env var:

   ```sh
   cp resources/prompts/course_outline/v1.prompt.txt resources/prompts/course_outline/v2.prompt.txt
   # edit v2 to taste
   echo "COURSE_OUTLINE_PROMPT_VERSION=2" >> .env   # or change the existing line
   uv run python -m coursesmith.hello "..."
   ```

3. **Missing env var fails loudly** ā€” unset `LITELLM_API_KEY`, run the CLI,
   observe a `pydantic_core._pydantic_core.ValidationError` naming the missing
   field rather than a confusing downstream auth error.

4. **Tests pass**:

   ```sh
   uv run pytest
   ```
