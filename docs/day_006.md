# Day 6 — Retries, timeouts & error handling

**Objective.** Make a single LLM call survive transient failures.

**Outcome.** Calls retry with backoff on 429/5xx, time out cleanly, and
surface a typed error to the caller.

## What we built

- **`LlmPort`** at `coursesmith/use_cases/shared/ports/llm_port.py` — a small
  abstract interface with `complete(messages, response_format)` and
  `stream(messages)`, plus a typed exception hierarchy: `LlmError` (base),
  `LlmTimeoutError`, `LlmRateLimitError`. The use case now depends on this
  port instead of importing LiteLLM directly, so retry/timeout policy lives
  in infrastructure where it belongs.
- **`LiteLlmAdapter`** at
  `coursesmith/infrastructure/shared/adapters/lite_llm_adapter.py` — the
  LiteLLM-backed implementation. Uses `litellm.router.Router` with
  `num_retries` and `timeout` so transient 429/5xx responses retry with
  backoff for free. Catches `litellm.exceptions.Timeout` /
  `RateLimitError` and re-raises as the port's typed exceptions (with
  `raise ... from e` so the original traceback is preserved as `__cause__`).
- **Streaming errors translated too.** `acompletion(stream=True)` returns the
  iterator quickly — errors actually fire during `async for` chunk iteration.
  The adapter's `stream` is therefore implemented as an async generator that
  wraps the iteration in the same try/except, so a timeout (or rate-limit)
  mid-stream surfaces as `LlmTimeoutError` / `LlmRateLimitError` to the
  caller, not a raw LiteLLM exception.
- **FastAPI exception handlers** in `coursesmith/app.py` map the typed
  port exceptions to proper HTTP responses: `LlmTimeoutError → 504`,
  `LlmRateLimitError → 429`. Callers now get a JSON `{"detail": "..."}`
  body with a meaningful status code instead of a generic 500.
- **Two new settings** in `Settings` (`coursesmith/settings.py`):
  `LITELLM_RETRIES` and `LITELLM_TIMEOUT`. Wired through `get_service()` in
  the REST adapter so configuration stays in `.env`, not code.
- **Four adapter tests** at
  `tests/infrastructure/shared/adapters/test_lite_llm_adapter.py`. Patch
  `_router.acompletion` to raise — or to yield a stream that raises
  mid-iteration — and assert that the right port exception comes out.
  Covers `complete` + `stream` × `Timeout` + `RateLimitError`.

## Why this satisfies the spec

- **"Calls retry with backoff on 429/5xx."** `Router(..., num_retries=N)`
  handles this automatically — retries with exponential backoff on retriable
  status codes before re-raising.
- **"Time out cleanly."** `Router(..., timeout=N)` enforces a per-call
  timeout; on expiry LiteLLM raises `Timeout`, which the adapter translates
  to `LlmTimeoutError`. No half-open requests, no hangs.
- **"Surface a typed error to the caller."** Use cases see `LlmError`
  subclasses, not LiteLLM internals. HTTP callers see 504/429 with a JSON
  body — no leaking stack traces, no generic 500s.

## How to verify

1. **Tests** — the adapter's translation behavior is covered:

   ```sh
   uv run pytest tests/infrastructure/shared/adapters/test_lite_llm_adapter.py -v
   ```

   Four passing tests prove that both `complete` and `stream` translate
   `Timeout`/`RateLimitError` to the port's exception types, including the
   tricky mid-stream case.

2. **Manual smoke test** — set an absurdly short timeout in `.env`:

   ```sh
   LITELLM_TIMEOUT=1
   LITELLM_RETRIES=0
   ```

   Restart the server, hit `POST /courses`, and you should see a `504` JSON
   response (`{"detail": "LLM call timed out after 1s"}`) instead of a 500
   with a LiteLLM stack trace.

3. **Full gate run**:

   ```sh
   uv run pre-commit run --all-files
   uv run pytest
   ```

## Design notes

- **Why a port at all?** Without the port, retry policy, timeouts, and
  provider-specific exceptions would leak into `CourseOutlineService` —
  which is application logic, not infrastructure. The port draws the line:
  the service speaks in domain terms (`LlmTimeoutError`), the adapter speaks
  in LiteLLM terms (`litellm.exceptions.Timeout`).
- **Why `Router` instead of `acompletion` + `num_retries=`?** `acompletion`
  also accepts `num_retries`, but `Router` is the path that will accept
  fallbacks and load-balancing later (Day 7+ material) without another
  rewrite. Same cost today, no migration tomorrow.
- **Streaming retries are partial by design.** Router can retry the *initial*
  connection but not resume mid-stream — once tokens have started flowing,
  a transient failure aborts the stream. The adapter at least guarantees the
  resulting error is *typed*, which is all the use case can react to.
- **`messages` is `list[dict[str, str]]` at the port boundary, `Any` at the
  LiteLLM call.** Plain dicts are what the OpenAI/Chat-Completions wire
  format wants, and what every adapter we'd ever write will accept. Casting
  at the boundary keeps the port stack-agnostic without taking on LiteLLM's
  typed message classes.
- **`response_format: type[T]`, not `T`.** We're passing a Pydantic *class*
  for structured-output validation, not an instance. The corrected typing
  catches misuse at the call site.
