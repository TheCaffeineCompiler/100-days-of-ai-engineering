# Day 5 — Streaming responses

**Objective.** Stream tokens to the client instead of waiting for the full
completion.

**Outcome.** A `POST /courses/stream` endpoint emits server-sent events; text
arrives incrementally in `curl -N`.

## What we built

- **`CourseOutlineService.create_stream(topic) -> AsyncGenerator[str, None]`** —
  async generator that calls `litellm.acompletion(..., stream=True)` and
  re-yields each chunk's `delta.content`. Filters out `None` deltas (LiteLLM
  emits an empty `delta` on the first and last chunks for role / finish-reason
  bookkeeping), so the wire only carries real content tokens.
- **`POST /courses/stream` endpoint** at
  `coursesmith/infrastructure/adapters/inbound/rest/create_course_outline_adapter.py`.
  Uses FastAPI's native SSE support: `response_class=EventSourceResponse` on a
  path operation that `yield`s `ServerSentEvent` instances. Each token becomes
  an `event: token / data: "<token>"` line; a final `event: done / data:
  [DONE]` marker tells the client the stream is over.
- **Streaming endpoint test** at
  `tests/infrastructure/adapters/inbound/rest/test_create_course_outline_stream.py`.
  Uses `app.dependency_overrides[get_service]` to swap in a fake whose
  `create_stream` is a plain async generator yielding two canned tokens.
  Asserts:
  - response is `text/event-stream`,
  - body contains the expected `event: token` lines with JSON-encoded
    `data:` payloads,
  - body contains the `event: done` / `data: [DONE]` terminator,
  - tokens appear before the terminator (order is preserved).

  No LLM calls, no env vars, runs in milliseconds.

## Why this satisfies the spec

- **"Stream tokens to the client instead of waiting for the full
  completion."** Tokens are forwarded as they arrive from the provider; the
  endpoint does not await the full response before responding.
- **"`/courses/stream` endpoint emits server-sent events."** Returns
  `text/event-stream`, framed as proper SSE (`event:` / `data:` per RFC).
- **"Text arrive incrementally in `curl -N`."** Verified manually: the curl
  client prints tokens one event at a time over a few seconds.

## How to verify

1. **Start the API**:

   ```sh
   uv run fastapi dev coursesmith/app.py
   ```

2. **Hit the stream with `curl -N`** so curl doesn't buffer:

   ```sh
   curl -N -X POST http://localhost:8000/courses/stream \
        -H 'Content-Type: application/json' \
        -d '{"topic": "Two-day intro to git"}'
   ```

   You should see something like:

   ```
   event: token
   data: "Course"

   event: token
   data: " Outline"

   event: token
   data: ":"

   ...

   event: done
   data: [DONE]
   ```

   Each `data:` payload is JSON-encoded (the model field is `Any` and FastAPI
   JSON-encodes everything that isn't `raw_data`), so string tokens come out
   quoted. The `[DONE]` terminator uses `raw_data=` so it appears as a plain
   literal.

3. **Non-streaming endpoint still works**:

   ```sh
   curl -X POST http://localhost:8000/courses \
        -H 'Content-Type: application/json' \
        -d '{"topic": "..."}' | jq .
   ```

4. **Tests**:

   ```sh
   uv run pytest
   ```

   Three tests: prompts adapter, non-stream endpoint, stream endpoint.

## Design notes

- **Streaming uses the same registered prompt as the non-streaming path.** No
  per-method prompt suffix. If a stream-specific prompt is ever needed (e.g.
  "give a markdown outline I can render as it arrives"), it should land as a
  new entry in the prompt registry, not as an inline f-string.
- **No `response_format=CourseOutline` on the stream call.** Structured-output
  enforcement requires the full response; it can't apply per-chunk. The
  streaming path therefore returns raw text tokens with no schema guarantee
  — that's the deliberate trade-off the spec asks for.
- **Why this is `POST` and not `GET`.** Native browser `EventSource` only
  speaks `GET`, but every other SSE client (curl, custom JS with `fetch +
  ReadableStream`, typed SDKs) handles `POST` fine, and `POST` keeps the
  request body in line with the non-streaming sibling endpoint.
