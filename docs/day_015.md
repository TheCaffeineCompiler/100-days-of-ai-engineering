# Day 15 — The agent loop (think → act → observe)

**Objective.** Wrap tool calling in a loop that runs until the model
stops asking for tools.

**Outcome.** A reusable `Agent.run(...)` keeps executing tool calls
and re-prompting until the model returns a final answer.
`CourseOutlineService` is now a thin caller of `Agent`; the same agent
also drives the streaming endpoint.

## What we built

- **`Agent` and `AgentTool` extracted into `use_cases/shared/agents/`.**
  The Day 13/14 loop lived inside `CourseOutlineService.create()`.
  This day pulls the *protocol* (call → if tool_use: run+append, repeat
  → else: return) into a reusable class and pushes the *tool catalogue*
  behind an ABC. Future use cases can now mount their own tools onto
  the same loop.

- **`AgentTool[TParams]` is a generic template-method ABC.** Subclasses
  declare the small surface — `name()`, `description()`, `params_cls()`,
  `_execute(params)` — and the ABC owns the duplicated work:

  ```python
  def get_schema(self) -> dict[str, Any]:
      return {"type": "function", "function": {
          "name": self.name(),
          "description": self.description(),
          "parameters": self.params_cls().model_json_schema(),
      }}

  async def execute(self, args: dict[str, Any]) -> Any:
      return await self._execute(self.params_cls().model_validate(args))
  ```

  Boundary validation is centralised. The LLM emitting an extra or
  malformed argument now surfaces as a `pydantic.ValidationError` from
  the ABC, not as a silent `KeyError` deep in a handler's `str.format`.

- **The three Day 14 tools became classes.**
  - `CreateTitleTool(AgentTool[CreateTitleParams])`
  - `CreateScheduleTool(AgentTool[CreateScheduleParams])`
  - `ReviewCourseTool(AgentTool[ReviewCourseParams])`

  Each tool carries its own `(prompts_name, prompts_version)` and an
  `LlmPort`, and uses the inherited `execute()` / `get_schema()` from
  the ABC. `ReviewCourseTool` additionally takes a `response_type`
  constructor argument since its inner LLM call is the one place we
  want structured output (the reviewed JSON is the final shape).

- **`get_current_time_tool` was removed.** Day 11/12/13's example tool
  no longer earned its registry slot once the planner workflow took
  over; deleting it (and its test) keeps the surface honest. Schema-shape
  coverage is restored by the three new tool tests.

- **`Agent.run(prompt_name, prompt_version, tools, prompt_params, response_format=None, max_steps=5)`.**
  Builds the user message from a versioned prompt template, then runs
  the loop. Returns the model's final text. Dispatch lives inside the
  agent — for each requested tool call we walk `tools` once,
  `if tool.get_name() == name: await tool.execute(params)`.

- **`AgentLoopExhaustedError`.** When `max_steps` runs out we raise a
  typed exception carrying the *original* budget (not the post-decrement
  zero). The REST layer in `app.py` translates it to **502 Bad Gateway**
  — the upstream model failed to make progress, semantically an
  upstream failure.

- **`Agent.stream(...)` keeps the same loop shape but switches modes on
  the final iteration.** Tool rounds use `complete`; the moment we see
  `tool_calls is None`, we re-issue the same `messages` through
  `LlmPort.stream` and yield each chunk's content. The service forwards
  the agent's tokens straight to SSE — no more LiteLLM-chunk unpacking
  in the service.

- **Composition root now wires each tool as its own
  `Depends`-target.** `get_agent`, `get_create_title_tool`,
  `get_create_schedule_tool`, `get_review_course_tool`,
  `get_service_tools` (a list builder), and `get_service` (which takes
  agent + tools). Every component still resolves to a lazy module-level
  singleton — request-scoped state lives in structlog contextvars, not
  in these instances.

- **`CourseOutlineService` shrank from ~60 lines to ~30.** Both
  `create` and `create_stream` are now thin pass-throughs around
  `Agent.run` / `Agent.stream`. The service still owns the
  `course_outline` prompt name and the JSON-validation step; everything
  else delegates.

## Why streaming had to be redesigned, not ported

Day 5's streaming endpoint streamed a single LLM call. The agent loop
makes *multiple* calls per request; most return tool_use blocks with no
user-visible content. The naive port — "stream every call" — would emit
nothing for the tool-use rounds and then a token stream for the final
answer, with no signal to the client that anything was happening in
between.

Three roads from here:

- **Option A (taken).** Run tool rounds non-streaming via
  `LlmPort.complete`; on the final iteration re-issue the messages
  through `LlmPort.stream` and forward chunks. Costs one extra LLM call
  per stream (the discovery call whose response we discard), but the
  agent's loop logic stays single-source.
- **Option B (not taken).** Stream from the start; assemble chunks with
  `litellm.stream_chunk_builder` to detect tool_calls; buffer tokens
  until we know we're in the text branch. More efficient, but the
  buffering defeats the latency win for the first few tokens and the
  interleaving is fiddly.
- **Option C (not taken).** Delete `/courses/stream`. Honest, but the
  endpoint is a published Day-5 contract; an agent-shaped successor is
  better than a hole.

A is the cheapest move that keeps the Day-5 contract working under the
new shape. B is where this should land if streaming becomes a focus
day.

## Tests

New under `tests/use_cases/shared/agents/`:

- **`test_agent.py`** covers `run` (no-tool path, one-round tool path,
  multi-tool dispatch-by-name, `max_steps` exhaustion with the original
  budget in the error message) and `stream` (tokens after no tool
  round, empty-delta filtering, tool round followed by streamed final
  answer, exhaustion).
- **`test_agent_tool.py`** covers the ABC's template methods —
  `get_schema()` builds the OpenAI shape from `name`/`description`/
  `params_cls`, `execute()` validates and dispatches, defaults apply
  for optional fields, missing/wrong-typed fields raise
  `ValidationError`.

New under `tests/use_cases/create_course_outline/tools/`:

- **`test_create_title_tool.py`** — schema shape +
  `response_format=None` on the inner call.
- **`test_create_schedule_tool.py`** — same, and asserts the schedule
  tool does *not* force the final `CourseOutline` shape on its
  intermediate output.
- **`test_review_course_tool.py`** — schema shape, the inner call uses
  `response_format=CourseOutline`, and a regression test that
  `content` is a required field with a description, not a default
  placeholder string.

The integration test was updated to construct the service with the new
`(agent, tools, prompt_version)` signature; the stub LLM returns no
tool calls, so an empty tools list is sufficient.

## What stayed weird

- **The discovery-then-re-issue extra call in `Agent.stream`.** Real
  cost per streaming request. Acceptable trade for code simplicity;
  revisit if streaming traffic grows.
- **`response_format` is silently dropped on the streaming path.** Same
  Day-5 caveat — structured output and streaming compete; we keep
  streaming and lose schema enforcement. The service's
  `create_stream` doesn't validate to `CourseOutline`.
- **Three module-level singletons hold one shared `LlmPort` instance.**
  Fine because the port reads request-scoped `request_id` from
  structlog contextvars at call time, not at construction. Worth
  remembering if anything in the LLM port ever takes per-request state
  by constructor.
- **`CreateTitleTool` and `CreateScheduleTool` are now structural
  near-duplicates.** Both wrap a prompt + a no-`response_format` LLM
  call. Resist generalising until a third near-duplicate appears.
- **`ReviewCourseTool.response_type: T` uses a `TypeVar("T")` that
  isn't tied to the class generic.** Functionally
  `type[BaseModel] | None` would be more honest. Cosmetic.
- **`prompts_name` / `prompts_version` constructor kwargs.** Singular
  reads better — each tool consumes one prompt, not many. Defer to a
  rename pass.

## How to verify

1. **Start the API**:

   ```sh
   uv run fastapi dev coursesmith/app.py
   ```

2. **Hit the JSON endpoint**:

   ```sh
   curl -X POST http://localhost:8000/courses \
        -H 'Content-Type: application/json' \
        -d '{"topic": "AI engineering for backend developers"}'
   ```

   Logs should show three `executing_tool` lines (`create_title`,
   `create_schedule`, `review_course`) followed by a final
   non-tool-call response. HTTP response parses as `CourseOutline`.

3. **Hit the streaming endpoint**:

   ```sh
   curl -N -X POST http://localhost:8000/courses/stream \
        -H 'Content-Type: application/json' \
        -d '{"topic": "AI engineering for backend developers"}'
   ```

   The tool rounds run silently; then `event: token` SSE frames
   start arriving; finally `event: done` with `data: [DONE]`.

4. **Run the suite**:

   ```sh
   uv run pytest
   ```

   32+ tests pass, including the new `tests/use_cases/shared/agents/`
   files and the updated integration test.
