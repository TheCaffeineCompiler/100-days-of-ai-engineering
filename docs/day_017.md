# Day 17 — Parallel tool calls

**Objective.** Execute multiple tool calls from one model turn
concurrently.

**Outcome.** When the model emits more than one `tool_call` in a
single assistant turn, the agent dispatches all of them through
`asyncio.gather` so they run in parallel. A failure in one tool no
longer cancels its siblings, and the model always receives one
`tool` message per `tool_call_id` it requested — including for
unknown tools or tools that raise. The course-outline workflow now
has two genuinely parallelizable siblings (`create_daily_outline`
and `create_daily_quiz`) that both consume the schedule, so the
parallel path is exercised end to end.

## What we built

- **`asyncio.gather` over the assistant's `tool_calls`.** The
  multi-call branch in `Agent._handle_tool_calls(...)` schedules one
  `_handle_tool_call(tc, tools)` coroutine per requested call and
  awaits them together. Two slow tools requested in the same turn
  now overlap instead of serializing.

- **Per-call error envelopes (the real fix).** First draft of this
  day flipped on `gather(..., return_exceptions=True)`, which kept
  one failure from cancelling its siblings — but then appended the
  `Exception` objects straight into `messages`. The next LLM call
  would have been rejected for two reasons: the message isn't a
  dict, and the model is missing a `tool` response for the failed
  `tool_call_id`. OpenAI/Anthropic both require one tool message per
  requested call.

  The shape that actually works: each `_handle_tool_call` is
  responsible for *always* returning a valid tool message, even on
  failure. We do that with a try/except inside the coroutine and a
  small helper:
  ```python
  @staticmethod
  def _tool_error_message(tc, name, error):
      return {
          "tool_call_id": tc.id,
          "role": "tool",
          "name": name,
          "content": json.dumps({"error": error}),
      }
  ```
  Same envelope is used for the unknown-tool case (previously
  returned `{}` with no `tool_call_id` — same latent bug). With this
  in place, `return_exceptions=True` became dead weight and was
  removed.

- **Two parallelizable siblings in the course-outline workflow.**
  `create_daily_outline` and `create_daily_quiz` both take a
  `Schedule` as input and produce independent artifacts. The v3
  prompt for `course_outline` explicitly tells the model to call
  `create_schedule` first, then both daily-content tools, then
  `review_course`. The model is free to request the two daily-content
  calls in the same turn — the agent loop now actually overlaps
  them.

  `CourseOutline` gained two fields to carry the output:
  ```python
  class CourseOutline(BaseModel):
      title: str
      day_items: list[DayItem]
      daily_outlines: list[str]
      daily_quizzes: list[DailyQuiz]
  ```

- **Composition wired through.** `get_create_daily_outline_tool` and
  `get_create_daily_quiz_tool` are added to the composition root and
  to `get_service_tools(...)`; two new prompt-version settings cover
  them. Prompt templates land as
  `resources/prompts/create_daily_outline/v1.prompt.txt` and
  `resources/prompts/create_daily_quiz/v1.prompt.txt`. The
  schedule-tool prompt also bumped to v2 — the previous version
  produced free text, but the daily-content tools need a typed
  `Schedule` to work with.

## Why per-call envelopes, not `gather(return_exceptions=True)`

`return_exceptions=True` *looks* like the parallel-failure fix, and
it does prevent cross-cancellation. But the contract with the model
is per-tool-call, not per-batch: every `tool_call_id` the model
requested must come back with a `tool` message before the next
assistant turn. If one is missing, the API rejects the request
outright; you don't get to "let the model recover from a Python
exception" because the model never gets the request.

So the fix has to live one level down — inside the per-call
coroutine — where we still have the `tc.id` and can build a proper
envelope. The model sees `{"error": "..."}` as the tool's content,
which it *can* recover from (e.g. retry with different args, or
proceed without that tool's output). The gather is then a pure
fan-out primitive with nothing to special-case.

## What stayed weird

- **Error envelope is `{"error": "..."}` JSON, not a richer shape.**
  Enough for the model to notice "this tool failed" and proceed; not
  enough to distinguish "unknown tool" from "tool raised" from "bad
  arguments." If we ever want to drive retries differently per
  category, the envelope can grow a `kind` field. Not today.
- **`Agent` still doesn't tell the model "use these tools in
  parallel."** That's purely a prompt-engineering and tool-design
  concern — the loop is willing to fan out the moment the model
  emits multiple `tool_calls`, but whether it *does* depends on the
  prompt and the shape of the tools. The v3 outline prompt nudges
  it; we're not measuring how often the model actually fans out
  versus serializes.
- **No backpressure / per-call concurrency cap.** If a future turn
  requested 50 tool calls at once, all 50 would launch immediately.
  Acceptable today (the model rarely emits more than ~3 in one
  turn for this workflow); a semaphore inside `_handle_tool_calls`
  would cap it cheaply when we need it.
- **`CourseOutline` doesn't yet validate that
  `len(daily_outlines) == len(day_items) == len(daily_quizzes)`.**
  The model is asked to produce one of each per day; nothing in the
  type system enforces it. A `model_validator` could, but it would
  push back into the loop with a `ValidationError` rather than a
  graceful retry — not the right shape until the agent loop has a
  validation-retry strategy.

## Tests

`tests/use_cases/shared/agents/test_agent.py` gains three Day 17
tests and two helper tool stubs (`_SlowTool`, `_FailingTool`) plus a
`_tool_calls_response(...)` helper that shapes an assistant message
with several `tool_calls` in one turn:

- `test_run_executes_parallel_tool_calls_concurrently` — two slow
  tools (`asyncio.sleep(0.15)` each) requested in one turn. Asserts
  the total wall-clock for `agent.run(...)` stays under 0.25s.
  Serial execution would have been ~0.30s. This is the test that
  catches an accidental future refactor that drops the `gather`.
- `test_run_isolates_failing_tool_call_and_reports_error_to_model` —
  one good tool + one tool that raises `RuntimeError("kaboom")` in
  the same turn. Asserts both `tool_call_id`s appear in the next
  LLM message, the good tool's content is unchanged, and the failing
  one's content is `{"error": "...kaboom..."}` JSON.
- `test_run_reports_unknown_tool_as_error_tied_to_tool_call_id` —
  model requests a tool that isn't registered. Asserts the agent
  still produces a `tool` message tied to the requested `tool_call_id`
  with an error payload, instead of dropping the call silently.

Two existing fixtures needed updating for the new `CourseOutline`
shape (`daily_outlines` / `daily_quizzes` required):
`tests/integration/test_create_course_outline_endpoint.py` and
`tests/infrastructure/.../rest/test_create_course_outline_adapter.py`.
`test_create_schedule_tool.py` was asserting
`response_format is None` (legacy: schedule was free text) — flipped
to assert `response_format is Schedule` now that the schedule tool
emits a typed `Schedule` so the daily-content tools can consume it
downstream.

1. **Run the gates**:

   ```sh
   uv run ruff check coursesmith tests
   uv run mypy coursesmith
   uv run pytest
   ```
   All three clean (ruff: 0 errors; mypy: 0 issues; pytest: 37
   passed).

2. **See parallel execution end to end** with structured logs:

   ```sh
   uv run fastapi dev coursesmith/app.py
   curl -X POST http://localhost:8000/courses -H 'content-type: application/json' \
        -d '{"topic": "git fundamentals"}'
   ```
   Two `executing_tool` log lines for `create_daily_outline` and
   `create_daily_quiz` should appear in the same loop iteration —
   timestamps overlap rather than chain.

3. **Force a per-call failure** by temporarily renaming a tool's
   `name()` so the model requests a name nothing matches. The next
   assistant turn should still succeed — the model receives an
   `{"error": "unknown tool: ..."}` envelope tied to the requested
   `tool_call_id`.
