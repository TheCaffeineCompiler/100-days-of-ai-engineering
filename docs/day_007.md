# Day 7 — Config & secrets management

**Objective.** Centralize all configuration into one typed settings object.

**Outcome.** A `Settings` object loads from env/.env; no scattered
`os.getenv`; missing required keys fail fast at startup.

## What we built

Almost nothing new — the discipline from Days 1–6 had already delivered the
spec by accumulation. The only real change today was a one-line fix to make
"fail fast at startup" literally true, plus a leaf-level singleton to break
a budding import cycle.

- **`Settings` was already centralized.** Day 1 introduced
  `coursesmith/settings.py` as a `pydantic-settings` `BaseSettings`
  subclass with `env_file=".env"` and no defaults. Days 3, 6 added more
  fields (`course_outline_prompt_version`, `litellm_retries`,
  `litellm_timeout`) but never branched off into a second config source.
  `grep -r "os.getenv\|os.environ" coursesmith/` returns zero matches.
- **One instance, instantiated at import.** The single line added today:

  ```python
  # coursesmith/settings.py
  settings = Settings()
  ```

  This runs when `coursesmith.settings` is first imported — which happens
  transitively from `coursesmith.app` before `uvicorn` binds the port.
  Missing env vars raise `pydantic.ValidationError` at startup, not at the
  first request.
- **Consumers import the instance, not the class.** The REST adapter
  changed from `from coursesmith.settings import Settings` +
  `settings = Settings()` inside `get_service()`, to
  `from coursesmith.settings import settings`. One less call, one less
  source of "where did we initialize this?" confusion.

## Why this satisfies the spec

- **"Centralize all configuration into one typed settings object"** —
  `Settings` is the single config surface; all five fields
  (`litellm_model`, `litellm_api_key`, `litellm_retries`, `litellm_timeout`,
  `course_outline_prompt_version`) live there.
- **"Loads from env/.env"** — `SettingsConfigDict(env_file=".env",
  extra="ignore")` reads process env first, falls back to `.env`.
- **"No scattered `os.getenv`"** — verified by `grep`. Zero matches in
  `coursesmith/`.
- **"Missing required keys fail fast at startup"** — the module-level
  `settings = Settings()` runs at import, before uvicorn accepts requests.
  A missing key crashes the process, not the first HTTP call.

## How to verify

1. **Startup fails fast on missing config**:

   ```sh
   LITELLM_MODEL= uv run fastapi dev coursesmith/app.py
   ```

   The process exits with `pydantic_core._pydantic_core.ValidationError`
   before binding port 8000 — not after the first request.

2. **Single source of env reads**:

   ```sh
   grep -rn "os.getenv\|os.environ" coursesmith/
   ```

   No matches.

3. **Gates still green**:

   ```sh
   uv run pre-commit run --all-files
   uv run pytest
   ```

## Design notes

- **Why the singleton lives in `settings.py`, not `app.py`.** The natural
  instinct is to put `settings = Settings()` in the composition root
  (`app.py`), next to `app = FastAPI()`. But the moment a lower-level
  module needs the instance, it has to import it back from `app.py` —
  which `app.py` already imports through the router chain. That's a
  circular import. Keeping the singleton at a leaf module (`settings.py`)
  means nothing imports the composition root, and the cycle never forms.
- **Why not `Depends(get_settings)` instead?** FastAPI's DI is the textbook
  alternative — `get_settings()` returns the cached instance and route
  handlers ask for it via `Depends`. It's cleaner for tests that need to
  override config per-request. We don't have that need yet; a module-level
  singleton is two lines and zero ceremony. Worth revisiting the day a
  test wants to vary env vars between cases.
- **`Settings()` has no defaults — on purpose.** A missing required key
  should be a startup error, not a silent fallback to an empty string that
  surfaces three layers deep as a confusing 401 from the provider. The
  `pydantic-settings` `ValidationError` lists exactly which keys are
  missing.
