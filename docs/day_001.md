# Day 1 — Router-ready project skeleton

**Objective.** Stand up the repo and make one LLM call through a LiteLLM router
you can swap models on without code changes.

**Outcome.** `python -m coursesmith.hello "ping"` returns a completion;
changing `LITELLM_MODEL=` in `.env` switches provider with zero code edits.

## What we built

- **Project scaffolding** with `uv` and `pyproject.toml` targeting Python
  `>= 3.12`. Dependencies pinned: `litellm`, `python-dotenv`.
- **`coursesmith/` package** with an `__init__.py` so the module is runnable
  as `python -m coursesmith.hello`.
- **`coursesmith/hello.py`** — a thin CLI that:
  1. loads `.env` via `python-dotenv`,
  2. reads `LITELLM_MODEL` and `LITELLM_API_KEY` from the environment,
  3. takes a single prompt argument via `argparse`,
  4. calls `litellm.completion(model=..., api_key=..., messages=[...])`,
  5. prints `response.choices[0].message.content`.
- **`.env.sample`** committed as a template; **`.env`** gitignored so real
  keys never enter version control.
- **`README.md`** with setup, run, and provider-swap instructions.

## Why LiteLLM satisfies "router"

`litellm.completion()` is a provider-agnostic call. The `model` argument uses
a `provider/model` prefix (e.g. `openai/gpt-4o-mini`,
`anthropic/claude-haiku-4-5`) and LiteLLM dispatches to the right backend.
Swapping providers is therefore a pure config change — the Python code never
references a specific provider SDK.

## How to verify

1. Set `LITELLM_MODEL` and `LITELLM_API_KEY` for provider A in `.env`,
   run `uv run python -m coursesmith.hello "ping"`, see a completion.
2. Change `LITELLM_MODEL` and `LITELLM_API_KEY` to provider B in `.env`,
   re-run the same command, see a completion from the new provider.
3. No edits to any `.py` file between steps 1 and 2.
