# Day 14 — Multiple tools / tool selection

**Objective.** Let the model choose among several tools.

**Outcome.** Three new tools sit alongside `get_current_time` in the
registry; for a single topic the model picks
`create_title` → `create_schedule` → `review_course` and returns a
populated `CourseOutline`. The Day 13 agent loop is unchanged in shape —
it just has more tools to dispatch.

## What we built

- **Three new tools, one file each.** All three follow the same shape as
  `get_current_time`: a `<name>_schema()` and an async handler, sharing a
  single Pydantic `*Params` model that is the source of truth for both
  the wire-level schema and the handler's argument type.

  - `create_title_tool.py` — topic → course title. LLM call against
    `course_title/v1.prompt.txt`. Params: `CreateTitleParams(topic: str)`.
  - `create_schedule_tool.py` — title → multi-day schedule. LLM call
    against `course_schedule/v1.prompt.txt`. Params:
    `CreateScheduleParams(title: str)`.
  - `review_course_tool.py` — title + day_items → improved
    `CourseOutline`. LLM call against `review_course/v1.prompt.txt`,
    *with* `response_format=CourseOutline` set this time — the reviewed
    output is the final shape, so we want the model to honour the schema
    here even though the outer call doesn't.

- **Versioned prompts per tool.** Each tool owns its prompt name as a
  module-level `PROMPT_NAME` constant; the version comes from
  `settings.<tool>_prompt_version`. Same pattern as
  `CourseOutlineService.PROMPT_NAME` and the same `(name, version)`
  resolution through `PromptsPort`.

- **`Settings` gets three new vars.** `create_title_prompt_version`,
  `create_schedule_prompt_version`, `review_course_prompt_version` — all
  required, no defaults. `.env.sample` updated to match, and
  `COURSE_OUTLINE_PROMPT_VERSION` bumped to `2` since v1 doesn't know
  about the tools.

- **New `course_outline/v2.prompt.txt`.** Lists the three tools,
  describes when to call each, and prescribes the workflow:
  `create_title → create_schedule → review_course → return`. v1 stays on
  disk for rollback (`COURSE_OUTLINE_PROMPT_VERSION=1`).

- **Boundary validation in the registry.** `execute_tool` now parses
  each tool call's JSON args into the matching `*Params` model *at the
  boundary* before invoking the handler:

  ```python
  if name == "create_title":
      return await create_title(
          llm_port=llm_port,
          prompts_port=prompts_port,
          request=CreateTitleParams.model_validate(kwargs),
      )
  ```

  The handler receives a typed `request: CreateTitleParams` — no
  `**kwargs` leaking dicts into typed code. If the LLM emits malformed
  args, pydantic raises and the outer `except Exception` in
  `execute_tool` turns it into a string the model can read back and
  retry from.

- **Loop hygiene.** `tools = get_tools()` is hoisted out of the
  iteration. Each tool call is logged as `executing_tool` with
  `tool_name=...`. `max_steps` exhaustion now logs a warning instead of
  silently returning the sentinel.

## What "tool selection" actually exercises here

Honest read of Day 14: the v2 prompt scripts the workflow order
explicitly (`use A, then B, then C`). The model isn't really *choosing*
in the open-ended sense — it's following a numbered plan. What the
registry does prove is the wiring:

- 3+ schemas reach the model in a single call,
- the model emits the right tool *name* (not a hallucination),
- each call dispatches through the registry's `name == ...` branch and
  validates into the right `*Params` model,
- one round's output becomes the next round's argument (the title from
  `create_title` lands as the `title` field of `create_schedule`'s
  request).

Pure selection — drop the workflow from the system prompt, harden each
tool's `description` with pre/post-conditions, let the model decide — is
on the Day 15 (agentic loop) backlog.

## What stayed weird

- **Tools wrap LLM calls of their own.** `create_title`,
  `create_schedule`, `review_course` are not deterministic functions —
  they're sub-LLM calls. The outer model is acting as a planner over
  sub-LLMs rather than over plain functions. Valid pattern, but the cost
  shape changes: every outer round-trip now triggers an additional
  completion *inside* the tool. Visible in the per-request usage
  tracker.
- **`if/elif` dispatch in `execute_tool`.** Four branches now. The
  dict-of-handlers refactor still hasn't happened — Day 15's agent-loop
  rework is the obvious moment.
- **`response_format=CourseOutline` + `tools=[…]` on the outer call.**
  Same Day 13 caveat. While the model is in tool-use mode it doesn't
  honour `response_format`; the discriminator stays "tool_calls
  present?".
- **`messages.append(result.choices[0].message)` pushes a LiteLLM
  `Message` object into `list[dict[str, Any]]`.** Carried over from
  Day 13. LiteLLM swallows it; the type is a lie.
- **No new tests.** The new tools touch the LLM port; schema-shape tests
  in the spirit of `test_get_current_time_tool.py` are the cheap
  follow-up and not done yet.

## How to verify

1. **Start the API**:

   ```sh
   uv run fastapi dev coursesmith/app.py
   ```

2. **Send a topic** (no smuggled-in trigger needed — v2's workflow
   covers any topic):

   ```sh
   curl -X POST http://localhost:8000/courses \
        -H 'Content-Type: application/json' \
        -d '{"topic": "AI engineering for backend developers"}'
   ```

3. **Confirm in the server logs** three `executing_tool` lines in this
   order:

   ```
   tool_name=create_title
   tool_name=create_schedule
   tool_name=review_course
   ```

   And `litellm.acompletion ... 200 OK` lines for each outer round plus
   one inside each tool.

4. **Confirm in the HTTP response** that `title` and `day_items` are
   populated from `review_course`'s output (not the sentinel; not the
   raw `create_schedule` content).
