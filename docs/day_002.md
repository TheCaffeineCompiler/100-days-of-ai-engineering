# Day 2 тАФ Structured outputs with Pydantic

**Objective.** Force the model to return a validated `CourseOutline` object
instead of free text.

**Outcome.** A function returns a typed `CourseOutline` (title, list of `Day`
items); malformed model output raises a clear validation error.

## What we built

- **`coursesmith/use_cases/create_course_outline/models/course_outline.py`** тАФ
  two Pydantic v2 models:
  - `DayItem` with fields `day: int`, `content: str`, `tasks: list[str]`.
  - `CourseOutline` with fields `title: str`, `day_items: list[DayItem]`.
- **`coursesmith/use_cases/create_course_outline/course_outline_service.py`** тАФ
  `CourseOutlineService` wraps the LiteLLM call:
  - constructor takes `model` and `api_key` (plain strings, injected by the
    caller тАФ no env reads inside the service);
  - a module-level `_prompt` template asks the model to produce a multi-day
    outline for a given topic;
  - `create(title: str) -> CourseOutline` calls `litellm.completion(...)` with
    `response_format=CourseOutline`, then runs
    `CourseOutline.model_validate_json(...)` on the response content. If the
    payload doesn't match the schema, Pydantic's `ValidationError` propagates
    to the caller untouched тАФ that *is* the clear validation error the spec
    asks for.
- **`coursesmith/hello.py`** тАФ CLI stays a thin wrapper: load `.env`,
  instantiate the service from env vars, call `service.create(title)`, print.
- **Layered package structure** introduced under `coursesmith/use_cases/`. Each
  future use case gets its own folder with `models/` and a service module, so
  the package can grow day-by-day without each feature touching the others.
- **`pydantic`** added as an explicit dependency in `pyproject.toml` (no longer
  relying on it being a transitive of `litellm`).

## Why this satisfies the spec

- "**A function returns a typed `CourseOutline`**" тАФ `CourseOutlineService.create`
  has the signature `(title: str) -> CourseOutline`. Mypy in strict mode
  confirms the return type end-to-end.
- "**Malformed model output raises a clear validation error**" тАФ the service
  does not catch `ValidationError`. Pydantic's default message names the
  offending fields and the expected types, which surfaces directly to the
  caller / CLI traceback.
- LiteLLM's `response_format=<PydanticModel>` is the structured-output
  contract: providers that support JSON-schema-constrained generation
  (OpenAI, Anthropic, Gemini, тАж) are given the schema; the model returns JSON
  that should match. The post-call `model_validate_json` is the safety net for
  providers (or runs) that don't honour the contract perfectly.

## How to verify

1. **Happy path** тАФ returns a typed object with day items:

   ```sh
   uv run python -m coursesmith.hello "AI engineering for backend developers"
   ```

2. **Schema mismatch raises** тАФ bypass the LLM and validate bad JSON directly:

   ```sh
   uv run python -c "from coursesmith.use_cases.create_course_outline.models.course_outline import CourseOutline; CourseOutline.model_validate_json('{\"title\": 1}')"
   ```

   This should exit non-zero with a `pydantic_core._pydantic_core.ValidationError`
   that names `title` and explains why an `int` isn't a `str`.
