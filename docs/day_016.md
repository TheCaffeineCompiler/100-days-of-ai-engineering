# Day 16 — Stop conditions & max iterations

**Objective.** Prevent infinite/expensive loops.

**Outcome.** `Agent.run(...)` halts on a final answer, a step budget, a
cost budget, or a wall-clock budget — and reports *why* it stopped via
a typed `AgentResult` carried on the `AgentLoopExhaustedError` payload.
`Agent.stream(...)` enforces the same three budgets. The REST layer
turns the typed exception into a structured 502 with a `stop_reason`
field the client can read.

## What we built

- **`AgentResult` carries `stop_reason` + `result`.** Lifted from
  Hint 2 literally: instead of `Agent.run` returning a bare string,
  it returns
  ```python
  class AgentResult[T: BaseModel](BaseModel):
      stop_reason: str
      result: T | str | None
  ```
  Happy path returns `AgentResult(stop_reason="finished", result=...)`.
  Exhaustion paths raise `AgentLoopExhaustedError(AgentResult(...))`
  with the appropriate `stop_reason`. The service unwraps `.result` and
  validates it into `CourseOutline`.

- **Three budgets, one helper.** `Agent._check_stop_conditions(...)`
  runs at the top of every iteration *before* the LLM call:
  ```python
  if max_steps < 0: raise(too_many_steps)
  if self._get_costs_in_cents() >= max_costs_in_cents: raise(costs_too_high)
  if (time.perf_counter() - loop_start) >= max_time_in_sec: raise(timeout_exceeded)
  ```
  Both `run` and `stream` call the helper — no drift between the two
  loops.

- **Cost budget reuses Day 9's `UsageTracker`.** The agent reads the
  *cumulative* cost for the current `request_id` (from the structlog
  contextvar) and compares against the budget — no per-iteration delta
  bookkeeping, no off-by-one. `Agent.__init__` now takes
  `usage_tracker: UsageTracker`; the composition root wires it in.

- **`loop_start = time.perf_counter()` is set once, *before* the
  loop.** First draft of this day had `start = ...` reset on every
  iteration, which meant the wall-clock check measured only the
  microseconds between assignment and the check — never the cumulative
  budget. The check could in practice never fire. (See "What stayed
  weird" for the embarrassed-laugh memo.)

- **Typed exception payload, typed handler.**
  `AgentLoopExhaustedError(context: AgentResult[Any])` and:
  ```python
  @app.exception_handler(AgentLoopExhaustedError)
  async def agent_loop_exhausted(_request, exc):
      payload = exc.args[0]
      if isinstance(payload, AgentResult):
          return JSONResponse(
              status_code=502,
              content={"detail": "agent loop exhausted",
                       "stop_reason": payload.stop_reason},
          )
      return JSONResponse(status_code=502, content={"detail": str(exc)})
  ```
  Clients now see *why* the loop stopped without parsing a free-form
  message.

- **PEP 695 type parameters land.** `AgentTool[TParams: BaseModel]` and
  `AgentResult[T: BaseModel]` use the new syntax. Mypy clean across the
  package; ruff stopped nagging UP046.

## Why we raise on exhaustion (instead of returning AgentResult)

Hint 2 says "return a structured result with a stop_reason rather than
just the text." We took half of that — the happy path returns
`AgentResult`. But the exhaustion paths *raise*
`AgentLoopExhaustedError(AgentResult(...))` rather than returning the
result with a non-"finished" reason.

Two reasons:

1. The downstream service has no useful behavior on a partial result —
   it would just convert it to a 502 anyway. Raising at the boundary
   keeps the success path clean (`agent_result.result` is always a
   string when stop_reason is "finished", which lets the service
   `cast(str, ...)` and validate without conditional branching).
2. The REST handler still gets the `AgentResult` payload (via
   `exc.args[0]`) and can map `stop_reason` to status codes — the same
   information is available, just not through the return type.

If a future use case wants to *react* to budget exhaustion (e.g.
"keep the partial answer"), this collapses: drop the raise, return
`AgentResult` with the appropriate `stop_reason`. Callers then branch
on it.

## What stayed weird

- **`Agent` now carries `UsageTracker`** as a constructor dep purely
  for the cost-budget read. The agent doesn't *track* usage — that
  still happens in `LiteLlmAdapter` — it just reads the snapshot.
  Acceptable today; if a third "read-only consumer of usage" ever
  appears, a `CostBudgetReader` port would be the cleaner cut.
- **`Agent.stream` re-implements the loop.** Yesterday's note still
  applies: every change has to land twice, once for `run` and once for
  `stream`. The `_check_stop_conditions` helper made today's update
  cheap, but the loop bodies are still siblings, not extracted.
- **`AgentResult.result: T | str | None`** is still a bit aspirational —
  the agent only ever sets it to a `str` (the LLM's final content); the
  service does the JSON validation. The `T` parameter is honest now
  (PEP 695 generics), but unused by the happy path.
- **Single 502 for all three exhaustion reasons.** `timeout_exceeded`
  arguably wants 504, `costs_too_high` arguably wants 402 or 422. Held
  off until we actually have a client that distinguishes; the
  `stop_reason` body field already lets them.
- **Cost-budget read is cumulative, not delta.** Correct by design
  (compare absolute spend to absolute budget), but worth naming
  explicitly so future-me doesn't try to "fix" it.

## Tests

`tests/use_cases/shared/agents/test_agent.py` is rewritten:

- A `_FixedCostUsageTracker` (subclasses `UsageTracker`, returns a
  fixed cost regardless of request_id) lets us rig the cost budget
  without binding a contextvar.
- All prior `run`/`stream` tests updated to:
  - construct `Agent` with the new `usage_tracker` kwarg
  - assert on `result.stop_reason` / `result.result` instead of a bare
    string return
  - assert the exception's `args[0]` is an `AgentResult` with the
    expected `stop_reason`
- Three new tests for Day 16 specifically:
  - `test_run_raises_with_costs_too_high_when_cumulative_cost_exceeds_budget`
    (tracker returns 10¢, budget 5¢, no LLM call fires)
  - `test_run_raises_with_timeout_exceeded_when_wall_clock_budget_is_zero`
    (`max_time_in_sec=0.0` — the loop's first check trips)
  - `test_run_returns_agent_result_with_finished_stop_reason_and_content`
    (happy-path shape)

The integration test was updated to pass `usage_tracker=UsageTracker()`
to the agent built in `_build_service_with_stub_llm`.

## How to verify

1. **Start the API**:

   ```sh
   uv run fastapi dev coursesmith/app.py
   ```

2. **Force a step-budget exhaustion** by giving the model a
   tool-rich topic and shrinking `max_steps` (temporarily set
   `max_steps=1` in `course_outline_service.py`'s `agent.run(...)`
   call). The endpoint should return:
   ```json
   {"detail": "agent loop exhausted", "stop_reason": "too_many_steps"}
   ```
   with HTTP 502.

3. **Force a time-budget exhaustion** by setting `max_time_in_sec=0.0`
   in the same call. The response should now read
   `stop_reason: "timeout_exceeded"`.

4. **Run the gates**:

   ```sh
   uv run ruff check coursesmith tests
   uv run mypy coursesmith
   uv run pytest
   ```
   All four clean (ruff: 0 errors; mypy: 0 issues; pytest: 34 passed).
